# Module 06 — 專題交付、角色分工與 API 治理

## 模組目標

這一模組把 05 拉回專題啟動現場，讓學員知道專題失敗常不是技術不會，而是範圍不清、角色不明、API 契約無主。學完後，學員應能寫出一份有範圍、有里程碑、有風險的啟動草案。

## 情境說明

團隊在 kickoff 會議中只討論技術棧，卻沒有定義哪些頁面保留 JSP、哪些功能先 API 化、誰負責 API 契約、誰負責驗收。三週後就開始出現前後端欄位不一致與排程失控。

## 核心重點

- 專題範圍定義
- 角色分工
- API governance
- milestone 與風險
- 驗收方式

## 教學步驟

### Step 1：先定義專題要解哪個問題

題目若定不清楚，後面的頁面、API 與架構都會漂移。

### Step 1-1：範圍定義要具體到頁面與流程

很多團隊會說「我們做理賠平台」，但這其實太大。更好的定義方式是：

- 本期先做理賠進度查詢與補件狀態查詢
- 不含完整理賠申請建立
- 不含主管審批流程

這種寫法才真的能控制範圍。

### Step 2：定義哪些東西暫時不改

這是 legacy 專案最重要的一步。要先決定保留哪些頁面與流程。

### Step 2-1：什麼叫做「暫時不改」

不是口頭說保留，而是要明確寫進啟動文件：

- 哪些 JSP 先沿用
- 哪些 jQuery 頁面只做局部補強
- 哪些 API 暫不重構

這能有效降低 scope creep。

### Step 3：定義治理節點

至少要有：

- API 規格 owner
- 前端整合 owner
- 測試驗收 owner

### Step 3-1：API governance 不只是誰寫文件

真正的治理至少要回答：

- 誰能改 API 契約
- 改動如何通知前後端
- 驗收依據是哪一版文件
- 何時 freeze，何時能變更

### Step 3-2：里程碑與風險要一起管理

如果里程碑只有日期，沒有風險說明，專案很容易看起來有進度，實際上一直在重工。

建議至少列出：

- legacy 依賴風險
- API 欄位變更風險
- 跨組協作風險
- 驗收延誤風險

## 範例：最小 governance 表

| 項目 | Owner | 更新方式 | 驗收依據 |
|------|-------|----------|----------|
| API 規格 | Backend lead | PR + 文件更新 | OpenAPI / Postman |
| 前端欄位映射 | Frontend lead | 週會確認 | 測試頁 / 畫面驗收 |
| 整體驗收 | QA / instructor | 驗收清單 | 指定 milestone |

## 範例：兩頁內啟動摘要應包含什麼

- 題目與商業問題
- 本期範圍與不做項目
- 角色與責任
- API 與頁面切分
- 里程碑與風險

這樣的文件才真的能拿來協作，而不是只有作業格式。

## 常見錯誤

- kickoff 只談技術，不談商業範圍。
- 沒有人負責 API 契約。
- 每週都在改 scope，卻沒有 freeze 點。
- 所有人都以為自己理解相同需求，但其實沒有共同文件。
- 啟動文件沒有列出「本期不做」，導致後續範圍膨脹。

## 自我檢查清單

- 我能清楚寫出本期做什麼、不做什麼嗎？
- 我知道 API 契約誰負責、怎麼更新、怎麼驗收嗎？
- 我能指出這個專題最大的 legacy 風險在哪裡嗎？
- 我寫的啟動文件能真的拿來協作，而不是只交作業嗎？

## 練習題

1. 寫一份兩頁內的專題啟動摘要。
2. 定義一張 API governance 表。
3. 列出本期明確不做的功能與原因。
4. 寫一份里程碑與風險清單。

## 解題方向

1. 題目要能回答目標使用者、核心流程與成功條件。
2. governance 題要定義 owner、更新方式與驗收節點。
3. 不做項目題要能防止 scope creep，不是敷衍列空白。
4. 風險題要把 legacy 依賴與跨組協作列出來。

## 完整參考答案

標準答案重點：

- 交付失敗常是治理失敗，不只是技術失敗
- 專題啟動文件要能拿來協作，不只是作業格式

## 驗收標準

- 能寫出有範圍、有不做項目、有 owner 的啟動摘要
- 能定義最小 API governance 表
- 能提出里程碑與對應風險
- 能說明 legacy 專案為什麼更需要治理而不是更少治理
