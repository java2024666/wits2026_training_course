# Scenario 06 — 重複建立請求與 Idempotency

## 背景

前端因網路延遲重送了同一個建立理賠請求兩次，後端成功寫入兩筆相同資料。從 HTTP 角度來看每次都回 200，但業務上已經出錯。

## 任務

1. 說明這題為什麼不是單純前端按了兩次按鈕。
2. 提出商業鍵、request key 或狀態檢查等處理方向。
3. 寫出完整參考答案與驗證方式。

## 解題方向

- 要把 HTTP 成功與業務安全分開看。
- idempotency 題要說明「如何辨識同一請求」。

## 完整參考答案

標準答案重點：

1. 先定義重複建立的判定方式。
2. 再設計去重或拒絕重送機制。
