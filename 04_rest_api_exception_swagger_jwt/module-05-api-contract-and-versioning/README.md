# Module 05 — API Contract、版本管理與可演進設計

## 模組目標

這一模組要補足 API 課程中很常被忽略的一塊：API 不是只要現在能用，還要能演進。學員要能理解 contract 是前後端與測試共同依賴的協議，一旦 DTO、路徑、欄位命名或狀態碼漂移，就會直接造成整合成本。

學完後，學員應能分辨哪些變更屬於安全擴充，哪些變更屬於 breaking change，並知道版本策略不是一個抽象名詞，而是與 API 生命周期直接相關的設計決策。

## 情境說明

團隊新增欄位時只改了 controller response，沒有更新 OpenAPI 與 Postman。前端仍照舊欄位解析，結果整合測試失敗。這不是文件小問題，而是 contract 管理失敗。

這一章的目標，就是讓學員從「API 可用」進一步提升到「API 可維護、可擴充、可交付」。

## 核心重點

- API contract 概念
- DTO 與文件同步
- pagination / filtering 設計
- idempotency 與重送安全
- 版本管理基本策略

## 教學步驟

### Step 1：定義 contract 是什麼

contract 至少包含：

- URL 與 method
- request / response 結構
- 狀態碼
- 錯誤格式

這裡要明確讓學員知道，contract 不是只看 JSON 欄位。只要以下任一項改了，都可能影響整合：

- path 改名
- method 改變
- 欄位型別改變
- 必填欄位增加
- 狀態碼語意改變

### Step 2：導入可演進設計

當需求開始加入條件查詢與分頁時，學員要知道這不是「多加幾個參數」而已，而是 API 介面設計的延伸。

例如理賠列表 API 一開始可能只有查全部，後續卻會逐漸加入：

- 狀態篩選
- 日期區間
- 分頁
- 排序

若這些規則沒有一開始就約定清楚，前後端很快就會各做各的。

### Step 3：分頁與查詢條件命名要一致

教學上可引導學員思考：

- `page`、`size`、`sort` 是否一致
- 篩選欄位是否能從名稱看出語意
- 查詢參數是否維持扁平與可讀

例如：

- `/api/claims?page=0&size=20&status=APPROVED`

通常會比 action 型查詢更容易理解。

### Step 4：談版本管理

不是所有變更都要立刻升到 v2，但 breaking change 必須有對應策略。

這裡要讓學員學會判斷：

- 新增非必填欄位，通常可視為相對安全的演進
- 刪除既有欄位、修改型別、改變錯誤格式，通常屬於 breaking change
- 路徑重命名或授權規則重大調整，也常需要版本或遷移策略

## idempotency 與重送安全

建立型 API 很容易遇到一個實務問題：前端 timeout 後重送，結果後端真的建立了兩筆資料。

教材要讓學員知道：

- 並不是只有資料庫唯一鍵才叫防重複
- 有些場景要靠業務鍵、requestId、idempotency key 或狀態檢查來避免重複建立

這一段很適合拿「理賠申請重送」作為例子。

## 常見錯誤

- 只改 DTO，不改 OpenAPI。
- 把 filter 參數設計成難以維護的 action 風格。
- 忽略重送請求造成的重複建立問題。
- 以為新增欄位永遠不會破壞前端。
- 覺得版本管理只要把 URL 改成 `/v2` 就算完成。
- 文件、Postman、實作三者長期漂移卻沒有發現。

## 自我檢查清單

- 我能清楚定義 API contract 包含哪些元素嗎？
- 我知道分頁與條件查詢如何保持一致命名嗎？
- 我能說明什麼情況可能需要版本升級嗎？
- 我能辨識什麼是 breaking change 嗎？
- 我知道建立 API 為什麼要考慮重送安全嗎？

## 練習題

1. 設計一條支援分頁與狀態篩選的理賠列表 API。
2. 設計一題 API 文件與 DTO 不一致的修正流程。
3. 說明建立 API 如何避免重送造成重複資料。
4. 舉出一個需要版本升級的 API 變更案例，說明原因。

## 解題方向

1. 分頁題要清楚定義 page、size、sort 與 filter。
2. contract drift 題要同時比對程式、OpenAPI、Postman。
3. idempotency 題要從商業流程與資料唯一性角度回答。
4. 版本題要判斷是否影響既有呼叫端，而不是只看開發端是否好改。

## 完整參考答案

標準答案重點：

- contract 是交付物，不是附屬文件
- 版本與欄位演進必須有一致規則
- 分頁、篩選、錯誤格式都屬於 contract 的一部分
- 可演進 API 的前提是先有穩定的命名與版本判斷標準
