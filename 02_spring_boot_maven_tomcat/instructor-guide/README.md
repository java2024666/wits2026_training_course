# 講師版教案

## 目的

這份教案不是給學員直接閱讀，而是幫講師在帶課時控制節奏、示範順序與提問方式，避免課程變成單純照著投影片念觀念。

本課的講師任務，不只是把四個 module 講完，而是要持續把學員拉回本課主軸：可部署、可追蹤、可交付。若課堂上只剩 API 細節或框架名詞，代表節奏已經偏掉。

## 建議授課節奏

1. 先用 10 分鐘說明企業後端交付為什麼不能只看程式碼。
2. 用 20 分鐘示範 Eclipse 匯入、Tomcat Runtime 建置與 WAR 部署。
3. 用 20 分鐘講 Maven、packaging、provided scope 與外部 Tomcat 的關係。
4. 用 25 分鐘拆解 Filter、Interceptor、AOP 的責任邊界。
5. 用 15 分鐘帶學員做 exercises 的第一題，讓學員看到真實故障排查順序。
6. 最後保留 10 分鐘做回顧與問答。

## 建議時數配置

若是半天課程，可使用以下配置：

- Module 01：20-25 分鐘
- Module 02：25-30 分鐘
- Module 03：25-30 分鐘
- Module 04：30-35 分鐘
- Exercises 導做：20-30 分鐘

若是一天版，可在每個模組之後插入 10-15 分鐘實作或小組討論。

## 講師操作原則

- 不要一開始就讓學員寫大量程式。
- 先讓學員知道系統是怎麼被部署、怎麼被追蹤、怎麼被交付。
- 每完成一段示範，就要求學員用一句話說明剛才解決的是哪一類問題。
- 遇到學員只想問 API 怎麼寫時，要拉回本課主軸：部署、打包、追蹤、稽核。

## 現場提問策略

每個模組至少要問一題「判斷題」，不要只問定義題：

- 這次錯誤是環境、打包、還是業務層？
- 這個欄位應該在哪一層建立？
- 這個設定若缺失，交接風險會是什麼？

這樣能把學員從背名詞拉回工程判斷。

## 建議授課切換點

講師在四個 module 中可用下列句型切換：

- 從 Module 01 進 Module 02：光知道要交付什麼還不夠，還要能在別人電腦上重建環境
- 從 Module 02 進 Module 03：環境建好後，接下來要理解產物為什麼能部署
- 從 Module 03 進 Module 04：系統能啟動還不夠，還要能追到每筆請求做了什麼

## Exercises 帶法建議

不建議一次把六題全部講完。較好的方式是：

1. 課中帶做 scenario-01 或 scenario-02
2. 課尾再讓學員分組拆 scenario-03 到 scenario-06
3. 要求每組用「問題層級、排查順序、修正內容、驗證證據」四段式回答

## 搭配文件

- demo-script.md：講師現場示範順序
- common-mistakes.md：學員常見錯誤與糾正方式
- module-walkthrough.md：逐模組教學重點與提問

## 講師驗收重點

本課帶完後，講師至少要確認學員能做到：

- 說出 WAR 與外部 Tomcat 的關係
- 說出 Maven packaging 與 provided scope 的部署意義
- 畫出 Filter / Interceptor / AOP 的基本時序
- 用正確順序回答至少一題故障排查題
