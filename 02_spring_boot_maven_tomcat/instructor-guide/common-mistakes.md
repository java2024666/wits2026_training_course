# 學員常見錯誤

## 錯誤 1：把課程理解成框架 API 教學

常見說法：我只要知道 Controller 怎麼寫就好。

講師修正方式：提醒學員本課的主題是可部署、可追蹤、可交付，而不是快速拼出功能。

## 錯誤 2：把 Filter、Interceptor、AOP 當成三選一

常見說法：有 AOP 了，為什麼還要 Interceptor？

講師修正方式：要求學員從請求生命週期與責任邊界回答，而不是從個人偏好回答。

## 錯誤 3：把 Maven 配置當成不重要的雜務

常見說法：這種設定檔不重要，能跑就好。

講師修正方式：直接連結到企業交付風險，說明同一份程式碼若打包方式錯誤，部署就會失敗。

## 錯誤 4：把稽核日誌等同 debug 訊息

常見說法：我有印 log 就算完成。

講師修正方式：要求學員列出 traceId、route、handler、status、duration，沒有這些欄位就不算稽核基線。

## 錯誤 5：覺得 `mvn package` 成功就等於部署完成

常見說法：WAR 都打出來了，應該不是打包問題。

講師修正方式：要求學員區分產物生成與外部容器部署是兩個不同驗證層次，並回到 WAR、provided、initializer、Tomcat 掛載這條因果鏈。

## 錯誤 6：把 Eclipse 匯入失敗當成程式碼錯誤

常見說法：專案好多紅線，應該是程式壞了。

講師修正方式：先要求學員檢查 JDK、Maven 辨識、project facets 與匯入方式，避免一開始就回頭改 Java 程式。

## 錯誤 7：把 Runtime 路徑與本機絕對路徑提交進版本庫

常見說法：我把所有設定都提交，別人應該更好跑。

講師修正方式：提醒學員共享設定與個人機器狀態不同，應提交可重建資訊，不應提交污染版本庫的本機絕對路徑。

## 錯誤 8：把 Filter、Interceptor、AOP 當成功能替代品

常見說法：既然 Interceptor 能記 log，就不用 Filter 了。

講師修正方式：要求學員從資訊出現時機回答，不要從個人偏好回答。traceId、route、duration 常常天然屬於不同層次。
