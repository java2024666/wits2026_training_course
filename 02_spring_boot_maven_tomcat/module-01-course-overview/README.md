# Module 01 — 課程總覽與版本控制基礎

## 模組目標

這一模組不是要把 Git 指令全部教完，而是要讓學員知道：企業後端專案的版本控制範圍，遠比 `src/main/java` 大很多。學完這一章後，學員應該能說明哪些檔案屬於正式交付物、哪些屬於個人環境資訊、哪些內容若沒有被版本控制，交接就一定會出問題。

更重要的是，要讓學員建立一個交付觀念：企業專案真正要被版本控制的，不只是程式碼，還包括能讓別人重建、部署、驗證與追溯這份專案所需的最小完整上下文。

## 開發痛點

後端工程師常把版本控制理解成「把程式碼推上 Git 就好」。但在金融與壽險產業，真正需要被追蹤的不是只有 Java 檔案，還包括：

- 專案使用的 JDK 版本
- Maven 打包方式
- 外部 Tomcat 版本
- 請求日誌攔截設計
- 環境建置說明

如果這些內容沒有被納入版本控制，團隊就會遇到以下問題：

1. 同一份程式碼，在不同工程師機器上啟動結果不同。
2. 交接時只剩口頭知識，沒有可追溯的建置依據。
3. 稽核要求追查某筆保單查詢或理賠申請當時跑的是哪個版本時，無法還原當時的狀態。

## 先建立一個正確心智模型

這一章要先把學員從「Git 是把程式丟上去」拉到「Git 是專案交付與追溯的邊界」。

教學上可以先建立三個分類：

- 業務程式碼
- 建置與部署設定
- 文件與交接資訊

只有把這三類一起看，學員才會理解為什麼這門課從版本控制開始，而不是直接從 Spring Boot 程式碼開始。

## 情境說明

假設你接手一個壽險內部系統專案，前一位工程師已經離職。現在團隊只留下：

- 一份可編譯的 Java 程式
- 幾句口頭說明「記得用 Tomcat 9」
- 一台本機曾經跑起來的電腦

這種情況下，哪怕程式碼本身沒壞，專案也很難真正交付。Module 01 的目標，就是建立這種交付意識，讓學員理解版本控制的邊界與最小完整交付單位。

## 技術解法

### 版本控制的最小交付單位

這門課要求學員提交的不只是 `src/main/java`，還包括：

- `pom.xml`
- `application.properties`
- Eclipse / Tomcat 操作文件
- WAR 打包所需設定
- 日誌攔截相關類別

這一段建議再補上一個判斷標準：

- 只要缺少某個檔案，下一位工程師就無法穩定重建、部署或驗證專案，那它就很可能屬於正式交付物的一部分

這裡要特別提醒學員：不是所有 IDE 產物都該提交，但足以幫助團隊重建專案的最小必要設定，通常應視為正式交付的一部分。

### Step 1：把「能重建」與「不該污染 Git」分開思考

很多新手最大的問題不是不會 commit，而是不知道該不該 commit 某些檔案。可以引導學員問兩個問題：

1. 這個檔案是否讓團隊能更穩定重建專案
2. 這個檔案是否只是我個人機器的暫時狀態

這個判斷方式會比單純背 `.gitignore` 更有用。

### 建議 Git 操作節奏

```bash
git init
git add README.md pom.xml src/
git commit -m "feat: initialize spring boot tomcat training project"

# 每完成一段可驗證功能，就提交一次
git add src/main/java src/main/resources
git commit -m "feat(logging): add request logging interceptor"
```

這裡要強調：commit 節奏不是越碎越好，也不是拖到最後一次全部上傳。好的節奏應對應「一段可以被驗證的改動」。

### 建議提交節點

1. 初始化專案骨架後提交一次。
2. 完成 Eclipse / Tomcat 部署設定後提交一次。
3. 完成 WAR 打包與 `provided` 調整後提交一次。
4. 完成 Filter / Interceptor / AOP 攔截後提交一次。
5. 完成 README 與 SETUP 文件後再提交一次。

### Step 2：教學上要把 commit 與可驗證狀態綁在一起

例如：

- 專案骨架能被 Maven 辨識
- Eclipse 可成功匯入
- WAR 可被 Tomcat 掛載
- 請求攔截日誌可看到 traceId

只要能被驗證，就值得成為一個提交點。

### 為什麼本課程從 Git 開始

因為企業交付不是「你本機跑得起來」就結束。你必須能向下一位工程師、SE、維運或稽核證明：

- 這個版本用什麼依賴打包
- 這個版本部署在哪一種 Tomcat
- 這個版本如何記錄保單查詢、理賠狀態與核保審核請求日誌

### Step 3：把版本控制與稽核追溯連起來

這門課不是單純教工具，而是讓學員知道：

- 沒有版本控制，就很難還原當時的程式與設定
- 沒有文件與 commit 紀錄，就很難回答「這個版本何時改了什麼」
- 在金融與壽險場景，這不只是方便性問題，而是交付與稽核問題

## 逐步操作

### Step 1：先列出正式交付物

請學員先回答：如果要把這個專案交給下一位工程師，至少哪些東西一定要存在？

建議答案至少包含：

- `pom.xml`
- `src/`
- 課程 README
- Eclipse / Tomcat 建置文件
- 打包與部署說明
- 攔截與日誌相關程式

這裡可以要求學員不要只列名單，而要逐一回答每個檔案的交付目的。例如：

- `pom.xml` 讓依賴與 packaging 可重建
- SETUP 文件讓下一位工程師知道如何掛外部 Tomcat
- 攔截相關類別讓請求稽核邏輯可被追蹤與維護

### Step 2：區分「應提交」與「不應提交」

應提交的通常是：

- 專案骨架
- 依賴描述
- 可重建環境的必要設定

不應提交的通常是：

- 個人 workspace metadata
- 本機絕對路徑
- 個人暫存檔與編譯輸出

這一步要補上常見灰區：

- 某些 Eclipse 專案描述檔可能需要保留
- 但工作區快取、server runtime 絕對路徑通常不應提交

也就是說，不能把所有 IDE 相關檔案一概視為垃圾；要看它是否有團隊共享價值。

### Step 3：練習寫可讀的 commit message

請學員針對以下情境各寫一條 commit message：

1. 新增外部 Tomcat WAR 部署設定
2. 新增理賠與保單請求的稽核攔截
3. 補上 Eclipse / Tomcat 建置說明

這裡建議再補一個判準：看到 commit message 後，團隊成員應至少知道：

- 改的是功能、設定還是文件
- 改動的主題是什麼
- 這次提交大概想解決什麼問題

### Step 4：把版本控制與交付風險連起來

要求學員回答：如果 `pom.xml` 沒被納入版本控制，會造成什麼後果？如果 SETUP 文件缺失，下一位工程師最可能卡在哪？

這一步是整章最重要的收束：要讓學員把 Git 與專案交付、環境重建、部署驗證連成一條線，而不是把版本控制看成獨立技能。

## 工程規範

### Commit Message 規範

採用 Conventional Commits：

- `feat(tomcat): configure external tomcat deployment`
- `feat(logging): add audit logging for policy and claim requests`
- `docs(setup): add eclipse tomcat runtime instructions`
- `fix(packaging): mark tomcat dependency as provided`

### 分支命名規範

- `feature/request-logging`
- `feature/war-packaging`
- `fix/tomcat-runtime-config`

### 交付規範

交付到 Git Repo 前至少要確認：

1. 專案可被他人 clone
2. `pom.xml` 可解出依賴
3. 文件可引導他人匯入 Eclipse 並部署
4. 稽核日誌功能可被驗證

這裡可以再補一句：若一份專案只能由原作者本人成功跑起來，那它就還沒有達到團隊交付標準。

## 常見錯誤

- 只提交 Java 原始碼，不提交打包與建置文件。
- 把 Git 當成備份工具，不當成交付追溯工具。
- commit message 寫成 `update`、`fix bug` 這種無法理解的描述。
- 不知道哪些 Eclipse 檔案是團隊需要、哪些只是個人環境產物。
- 以為 README 只是附加文件，不把它視為交付物。
- 程式能編譯就算完成，沒有檢查部署與重建流程是否可被他人重現。
- 把本機路徑、server 狀態或暫存輸出一併提交，污染版本庫。

## 自我檢查清單

- 我能說明這份專案的正式交付物有哪些嗎？
- 我能區分團隊應共享設定與個人本機設定嗎？
- 我寫的 commit message 是否讓別人看得懂我改了什麼、為什麼改？
- 我能回答「如果今天要交接，下一位工程師要靠哪些檔案重建環境」嗎？
- 我能說出哪一些 IDE 相關檔案其實有共享價值，哪一些只是個人狀態嗎？
- 我能把版本控制、交接、部署與稽核追溯串成同一件事來理解嗎？

## 練習題

1. 列出本課專案最小可交付清單。
2. 寫出 3 條符合 Conventional Commits 的 commit message。
3. 說明為什麼 Tomcat 版本、Maven 打包方式與 README 也屬於版本控制範圍。
4. 從交接角度，說明如果缺少 `pom.xml`、SETUP 文件、日誌攔截類別，各自會造成什麼風險。

## 練習解答方向

1. 清單題不要只寫 `src/`，要把 `pom.xml`、README、部署說明一起列入。
2. commit message 題要能看出功能、設定或文件的差異。
3. 範圍題要從「可重建、可追溯、可交接」三個角度回答。
4. 風險題要把「編譯、部署、驗證、稽核」四種後果拆開說明。

## 本模組驗收

- 能說明為什麼 Tomcat 版本與打包設定也屬於版本控制範圍
- 能寫出至少 3 個可讀的 commit message
- 能說明本週交付物不只是 Java 程式碼，而是一個可部署、可追溯的專案
- 能區分正式交付設定與個人本機設定
- 能從交接與稽核角度說明版本控制的實際價值
